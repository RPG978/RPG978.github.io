# Software Design and Engineering
## 1.	Briefly describe the artifact. What is it? When was it created?
The artifact I selected for this category is a text based adventure game originally created in IT 140: Introduction to Scripting. The original version, written in Python, was a procedural, single file program designed to demonstrate basic control flow, user input handling, and simple game logic. I created it early in my academic program, and at the time it met the assignment requirements but lacked structure, modularity, and long term maintainability.
For this enhancement, I translated the basic mechanics of the game into C++ and redesigned it using a clean, object oriented architecture. The new version introduces core classes such as GameEngine, Room, Player, and Puzzle, each with clearly defined responsibilities. This redesign lays the foundation for future extensibility, including data driven puzzle loading , subsystem expansion and more complex game mechanics.
## 2.	Justify the inclusion of the artifact in your ePortfolio. Why did you select this item? What specific components of the artifact showcase your skills and abilities in software development? How was the artifact improved?
I selected this artifact because it clearly demonstrates my growth as a software engineer and is deeply in line with my career goals. The enhanced C++ version showcases my ability to design maintainable systems, apply modern programming practices, and think architecturally rather than procedurally. A modular class design improves maintainability by separating responsibilities across well defined subsystems. For example, puzzle logic is now isolated in PuzzleEngine, making it easier to extend without modifying unrelated code. Improved gameplay behavior emerged naturally from the redesign. The engine now supports dynamic puzzle state, locked exits, and room specific interactions that were difficult or impossible to express cleanly in the original Python script. An improved input handling and defensive programming strengthen reliability and reflect a security aware mindset aligned with professional expectations. Clear documentation and architectural reasoning demonstrate my ability to communicate technical decisions effectively. My enhancements have transformed a simple script into a scalable codebase that resembles a real game engine skeleton.

## 4.	Reflect on the process of enhancing and modifying the artifact. What did you learn as you were creating it and improving it? What challenges did you face?
Enhancing this artifact required me to rethink the entire structure of the original program. Instead of simply rewriting Python logic in C++, I had to identify the underlying responsibilities in the game and design a system that could support them cleanly. This process taught me how to break down a problem into components, define clear interfaces, and anticipate future requirements.
One of the biggest challenges was ensuring that the architecture remained simple enough for the scope of the project while still demonstrating professional engineering discipline. I had to balance abstraction with practicality. Feedback encouraged me to clarify why C++ was the right choice and to explicitly incorporate secure input handling, which strengthened the final design and aligned it more closely with Outcome 5.
I also learned the importance of environment stability and reproducibility. During development, I encountered issues with Visual Studio project structure, Git repository configuration, and build artifacts. Solving these problems reinforced my understanding of how tools, folder structures, and version control interact, directly supporting Outcome 4.
To validate the enhanced version, I tested the game iteratively as each subsystem was introduced; verifying room navigation, puzzle triggers, locked exits, and database driven world loading. This incremental testing helped ensure that new features integrated cleanly and that the redesigned architecture behaved as intended.
Finally, I created an initial UML class diagram to outline the intended architecture before writing any C++ code. As development progressed, the UML naturally drifted from the original design. This drift reflected a deeper understanding of the system’s needs as I implemented the walking skeleton. The UML was not a static artifact; it evolved alongside the codebase and served as a living design tool throughout the enhancement process.
